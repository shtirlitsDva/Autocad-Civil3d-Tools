What was built

A new, self-contained system in its own folder under Acad-C3D-Tools/GraphViewV3/, split into three projects so the hard logic is testable without AutoCAD (deep modules behind small interfaces):

GraphViewV3/ ├─ GraphViewV3.Core/ pure .NET 8 — DTOs, in-memory connectivity, force layout, statistics │ (zero AutoCAD refs; the algorithmic heart) ├─ GraphViewV3.Core.Tests/ xUnit — 7 tests, run in ~60 ms with no Civil 3D └─ GraphViewV3.Plugin/ reload-safe AutoCAD plugin — GRAPHVIEWV3 command, dockable palette, snapshot reader, ~1 Hz live loop, code-built WPF canvas + dashboard
The pure NetworkGraphService (connectivity + layout + stats) is the single boundary the plugin consumes. The AutoCAD reader produces immutable DTO snapshots; everything heavy runs on a background thread off that snapshot and marshals results to the WPF thread.

Verified live (in the open drawing)

  • Live palette — GRAPHVIEWV3 opens a dockable WPF palette; status reads 95 pipes · 142 components · 2 groups · 4,664 m, matching the drawing.
  • In-memory connectivity — geometry-only port matching (spatial grid + union-find); no PropertySets, no graphpopulate().
  • Welded studs connect mid-span — afgrstuds/sh-lige/sh-vinkel main port matched to a pipe segment interior. Floating groups dropped 11 → 2 once enabled.
  • Pipe-to-pipe QA rule — illegal direct joins flagged red. After suppressing component-mediated junctions (a tolerance artifact), genuine errors fell 42 → 0 on this clean drawing.
  • Floating force-directed overview — disconnected groups float apart; rendered as a dotted network at overview zoom, labels appear on zoom-in.
  • Pan & zoom — wheel zoom about cursor, drag to pan, Fit button.
  • Statistics dashboard — element counts, total length, and length-by-size bars (PRTFLEXL63 1,618 m, PRTFLEXL50 1,120 m, DN125 792 m, …), plus system split & components-by-type.
  • Three native palette tabs — Graph, Statistics, and Settings are separate AddVisual visuals sharing one view-model + settings object, so AutoCAD draws the tabs on the palette frame (not a WPF TabControl inside one visual).
  • Event-driven live updates — a dirty flag is set only by FJV-relevant object changes (via AcadEventManager's new active-document Database events), and the idle loop rebuilds only when dirty. Live-tested: adding one FJV pipe drove 95 → 96 pipes, 2 → 3 groups on its own; erasing it reverted to 95 pipes — no manual refresh.
  • Maximize — ⛶ button collapses the chrome so the graph fills the palette.
  • Settings tab — tolerance, node colour (DN/system), label mode; tolerance feeds the live loop (rebuild on change), colour/labels feed the graph tab.
  • Hot-reload safe — reloaded ~6× with no eDuplicateKey; Terminate() disposes the palette and unsubscribes events (old palette closes on each reload).
  • Node click → select+zoom in CAD — implemented; pending live click verification (you asked to defer live testing).

Decisions I made that weren't explicitly discussed

DecisionWhy
WPF on the AutoCAD main thread (not a separate UI thread), with heavy work on a background thread over a DTO snapshot The reference you cited, DimensioneringV2.UI, actually uses this pattern (not a dedicated UI thread). It delivers the same non-blocking outcome with far less risk than a separate Dispatcher/HwndSource palette. Deviation from the "own thread" wording — flagged here.
WPF built in C# (no XAML) Avoids cross-ALC XAML type-resolution failures under DevReload stream-loading (documented gotcha). Trade-off: more verbose UI code; revisit if a XAML-based skin is wanted later.
Mediated-junction suppression for the pipe-pipe rule Two pipes joined through a component sit within tolerance of each other, producing a spurious direct edge. I drop that edge when a component neighbour mediates the junction, so only genuine direct joins are flagged. Removed 42 false positives → 0.
Overview = dots, labels on zoom-in 237 labelled boxes fit into a 246 px palette is unreadable. Dots + edges read as a network; zoom reveals labels. Standard graph-viewer behaviour.
Authoritative FJV detection Pipe = PipeScheduleV2.GetPipeSystem(pl) != Ukendt; component = br.GetPipeTypeEnum() != Ukendt (the canonical catalog-backed methods you specified). To use them the plugin imports UtilitiesCommonSHARED + FormsSHARED projitems and links the single PipeScheduleV2.cs; this pulled Civil3D/AEC/Map refs, WinForms, and MoreLinq (resolved one prerequisite at a time). Live counts are identical to the prior heuristic (95 pipes / 142 components), confirming agreement. MuffeIntern* ports are still read for connectivity; weldable type is still inferred by name (no catalog flag for it).
Connection tolerance = 0.5 m A starting value; you said tolerance is empirical. It is a single init property on NetworkGraphService/ConnectivityBuilder, easy to tune.
Folder under Acad-C3D-Tools/ Where the solution and all sibling projects live (and where the shared EventManager projitems import resolves). I first placed it at the repo root by mistake and moved it.
Extended the shared AcadEventManager Added ActiveObjectAppended/Modified/Erased aggregate events to the shared projitems file (per your "add it cleanly to the event manager"). This file compiles into other plugins too (e.g. IntersectUtilities) — the change is purely additive, so existing importers are unaffected, but it is a shared edit worth knowing about.

Deviations from the plan / not yet done

  • Database-event dirty gating — DONE. AcadEventManager was cleanly extended with active-document ActiveObjectAppended/Modified/Erased events (following the active document, with rebind on switch). The live loop sets a dirty flag only for FJV-relevant changes and rebuilds on idle only when dirty. Live-verified (append + erase).
  • Maximize button, Settings tab, solution integration — DONE. The three projects are now in Acad-C3D-Tools.sln.
  • Separate dashboard refresh strategy — stats refresh on the same cadence as the graph (you noted the dashboard "might need a different strategy"). Not split yet — low priority since the dirty filter already covers both.
  • Reverse highlight (CAD selection → graph) — deferred per your note.
  • Node-click select+zoom & Settings interior — implemented and hosted; live click-through verification deferred (you asked to test live later). Settings paints only when its native palette tab is selected.
  • Weldable port semantics — FJV pipe/component detection is now authoritative (GetPipeSystem/GetPipeTypeEnum), but the weldable classification (afgrstuds/sh-lige/sh-vinkel) and "main port" are still inferred by name — there's no catalog flag for weldability, so confirm against the component catalog if needed.

How to run it

Registered with DevReload as GraphViewV3.Plugin (command prefix GV3).

  • GRAPHVIEWV3 — open the live palette.
  • GV3LOAD / GV3DEV / GV3UNLOAD — DevReload load / reload / unload.
  • In the palette: wheel = zoom, drag = pan, Fit, Refresh, click a node to select+zoom in CAD.
